Overview

This project implements two statistical randomness tests from NIST SP 800-22 Rev 1a as both a sequential C++17 baseline and a SYCL/DPC++ parallel implementation targeting Intel CPU and GPU devices through Intel oneAPI.

⚗

Runs Test

Section 2.3 — Tests whether the number of runs (consecutive identical bits) is consistent with a truly random sequence.

📊

Longest Run of Ones

Section 2.4 — Tests whether the longest run of ones within fixed-size blocks matches the expected distribution.

⚡

SYCL Acceleration

Parallel kernel implementations using Intel oneAPI DPC++, targeting both CPU and GPU with a shared source code.

✔

NIST Validated

Results are verified against official NIST SP 800-22 reference test vectors, confirming numerical correctness.

Features

Dual implementation

Sequential C++17 baseline and SYCL parallel version running side-by-side for direct comparison.

Multi-backend GPU support

Targets Intel integrated/discrete GPUs, NVIDIA (nvptx64), and AMD (amdgcn) via SYCL backend flags.

Flexible input formats

Accepts ASCII '0'/'1' text files, packed binary files, or in-process generated random sequences.

CSV & text output

Results can be emitted as formatted text or CSV for downstream analysis and graphing.

Benchmark mode

Runs sequential and SYCL versions on the same input and reports speedup, throughput, and p-value agreement.

Tunable work-group size

Expose --workgroup flag (32 / 64 / 128 / 256 / 512) to find the optimal tile size for a given device.

Sequential-only mode

Build and run with any C++17 compiler — no SYCL runtime or GPU required.

CMake & Makefile builds

Both build systems are provided, with clear instructions for sourcing the oneAPI environment.

Prerequisites

SYCL build

  • Intel oneAPI Base Toolkit 2024.1+
  • icpx compiler with -fsycl support
  • Linux (Ubuntu 20.04+) or Windows WSL2
  • CMake 3.16+ (optional)

Sequential build

  • Any C++17 compiler: g++ 8+, clang++ 10+, or MSVC 2019+
  • No GPU or SYCL runtime required

NVIDIA GPU backend

  • Add to Makefile/CMake:
  • -fsycl-targets=nvptx64-nvidia-cuda

Algorithm Details

Runs Test — Sequential

O(n)

A single linear scan over the bit array counting transitions (positions where b[i] ≠ b[i-1]). The count is Vobs. The prerequisite check π ∈ (2/√n, 1−2/√n) is applied first; failure immediately sets SKIP.

P  =  erfc(|Vobs − 2nπ(1−π)| ⁄ (2√(2n) ⋅ π(1−π)))

Runs Test — SYCL Parallel

O(n/W)

The sequence is divided into chunks of 1 024 bits. Each work-item independently counts transitions within its chunk and records the left/right boundary bits. A host-side O(W) merge pass reconciles chunk boundaries. This achieves near-linear work with W-way parallelism.

Partition → Per-chunk count → Boundary merge → P-value

Longest Run — Sequential

O(n)

Scan through N blocks of M bits each. Within each block, track the current run length, updating the block maximum. Accumulate frequencies into histogram bins as defined by the NIST table for the chosen (K, M, N) parameter set.

P  =  igamc(K/2, χ²/2)

χ² = ∑i=0K (vi − Nπi)² / (Nπi)

Longest Run — SYCL Parallel

Embarrassingly parallel

One work-group is assigned per M-bit block. Within the group, work-items split M bits into segments and compute local longest runs plus prefix/suffix run lengths. A tree reduction in local (shared) memory correctly merges spans that cross segment boundaries to yield the block-global maximum.

Assign WG/block → Segment runs → Tree reduction → Histogram → P-value

Performance

Speedup is measured on an Intel UHD Graphics 770 (integrated GPU) against the sequential C++17 baseline. SYCL overhead is dominant for small sequences; speedup becomes significant at ≥1 M bits.

Key observations

  • For sequences < 10 K bits, SYCL device selection, buffer allocation, and kernel launch overhead exceeds computation time. Sequential is faster.
  • The Longest Run test benefits more from parallelism than the Runs test because it is embarrassingly parallel at the block level.
  • Optimal work-group size depends on the device — experiment with --workgroup 64|128|256|512.
  • Use --benchmark mode to measure both implementations on the same input and see the real-world speedup for your hardware.
